# IIS ASP.NET Core Module (Out-of-Process) 502 Reproduction

This repository provides a test suite to reproduce a critical issue where the **IIS ASP.NET Core Module (ANCM)** in **Out-of-Process mode** returns a **502 Bad Gateway** for specific request patterns, even though these requests are handled correctly in In-Process and Classic ASP modes.

## The Problem
The ANCM (Out-of-Process) fails to proxy requests to the backend Kestrel process under the following two conditions:
1.  **HTTP/2:** A `GET` request followed by a `DATA` frame.
2.  **HTTP/1.1:** A `GET` request with `Transfer-Encoding: chunked`.

### Key Observations:
*   **502 Origin:** The 502 error is generated by **IIS**, not the ASP.NET Core application.
*   **No Connection:** Wireshark captures confirm that the **.NET application process never receives any TCP connection** for these requests in Out-of-Process mode.
*   **Protocol Translation:**
  * In **Classic ASP** mode, both HTTP/2 and HTTP/1.1 (chunked) requests are correctly translated/received as `HTTP/1.1 + Transfer-Encoding: chunked`.
  * In **In-Process** mode, HTTP/2 translated/received as `HTTP/2 + Transfer-Encoding: chunked` (via `HttpContext.Request.Method`), HTTP/1.1 (chunked) requests are correctly translated/received as `HTTP/1.1 + Transfer-Encoding: chunked`.

## Comparison Matrix

| Client Request Type | IIS Mode | Result | App Received Request |
| :--- | :--- | :--- | :--- |
| **HTTP/2 GET + DATA** | Out-of-Process | ❌ **502** | *No Connection* |
| **HTTP/2 GET + DATA** | In-Process | ✅ 200 | `HttpContext.Request.Method` got `HTTP/2`, Chunked |
| **HTTP/2 GET + DATA** | Classic ASP | ✅ 200 | HTTP/1.1 Chunked |
| **HTTP/1.1 GET (Chunked)** | Out-of-Process | ❌ **502** | *No Connection* |
| **HTTP/1.1 GET (Chunked)** | In-Process | ✅ 200 | HTTP/1.1 Chunked |
| **HTTP/1.1 GET (Chunked)** | Classic ASP | ✅ 200 | HTTP/1.1 Chunked |

## Project Structure

```text
.
├── client/           # Go-based HTTP/2 & HTTP/1.1 Clients
├── golang/           # Reference Server (Verbose HTTP/2 logging + Header dump)
├── dotnet/           # ASP.NET Core 10 Web API (Target app)
└── asp/              # Classic ASP (.ashx) handler (Baseline comparison)
```

### Components
*   **`/client`**: Supports two modes:
    1.  HTTP/2 GET followed by a `DATA` frame.
    2.  HTTP/1.1 GET with `Transfer-Encoding: chunked`.
*   **`/golang`**: A reference server to verify that the client is sending frames correctly and to dump raw headers.
*   **`/dotnet`**: An ASP.NET Core 10 API with a header dump handler to verify successful processing.
*   **`/asp`**: A Classic ASP script used to prove that the IIS worker process (w3wp.exe) handles the request correctly independently of ANCM.

## Reproduction Steps

### 1. Environment Setup
*   IIS with ASP.NET Core Hosting Bundle installed.
*   Go installed (for client and reference server).

### 2. Deployment
*   **Site A (Out-of-Process):** Deploy the `.dotnet` project in Out-of-Process mode.
*   **Site B (In-Process):** Deploy the `.dotnet` project in In-Process mode.
*   **Site C (Classic ASP):** Deploy the `asp` project.

### 3. Test Execution & Verification
1.  **Test Out-of-Process:** Run the client against **Site A**.
    *   *Expected:* IIS returns `502`. Verify with Wireshark that no TCP connection is established to the .NET app.
2.  **Test In-Process:** Run the client against **Site B**.
    *   *Expected:* `200 OK`. Verify the dump shows `HTTP/1.1` and `chunked` encoding.
3.  **Test Classic ASP:** Run the client against **Site C**.
    *   *Expected:* `200 OK`. Verify the dump shows `HTTP/1.1` and `chunked` encoding.

---

## Debugging Evidence
To confirm the issue is within the ANCM module:
1.  **Wireshark:** Capture traffic between the Client, IIS and backend. In Out-of-Process mode, the connection to the backend port (which must be retrieved from the ASPNETCORE_PORT environment variable of the dotnet.exe process) is never initiated.
2.  **Golang Logs:** Use the `/golang` server to ensure the client's HTTP/2 frame sequence is standard and valid.
